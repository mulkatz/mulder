# Spec 49: `mulder show` command — terminal preview of layout.md

## 1. Objective

Add a `mulder show <source-id>` CLI command that reads the `layout.md` artifact produced by the Extract step (spec 48) and prints it to stdout with lightweight ANSI formatting. Makes the derived Markdown immediately consumable in the terminal without opening a file in an editor — the cheapest possible demoability win for the Extract pipeline stage while the real document viewer waits for M7-H11.

This is a focused stop-gap: a single-file CLI command (~200 lines) that uses the existing service abstraction, adds zero new dependencies, and will be naturally superseded once the M7 HTTP API + React viewer lands. It does not create architectural debt — it uses the same `loadConfig() → createServiceRegistry() → services.storage.download()` pattern as every other CLI command, and its formatter is a pure function of the input string.

**Spec refs:** §1 (CLI surface — where `show` slots in alongside `ingest`, `extract`, `query`), §2.2 (extract step produces `layout.md`), §4.5 (service abstraction — CLI commands call services, never direct GCS).

## 2. Boundaries

### In scope

- New CLI command `mulder show <source-id>`:
  - Positional arg: `<source-id>` (UUID of a previously-extracted source)
  - Option `--raw` — print plain Markdown, no ANSI colors (for piping/redirecting)
  - Option `--pager` — pipe formatted output through `$PAGER` (falls back to `less -R`, which preserves ANSI colors)
  - Option `--help` / `-h` — must work (auto-generated by Commander, explicitly asserted by QA)
- Implementation in `apps/cli/src/commands/show.ts`, one file, following the existing `registerXxxCommands(program: Command)` pattern used by every other CLI command.
- Registration in `apps/cli/src/index.ts` via `import { registerShowCommands } from './commands/show.js';` and a `registerShowCommands(program);` call placed alphabetically between `registerSegmentCommands` and `registerTaxonomyCommands`.
- **Hand-rolled ANSI formatter** using `chalk` (already a dep in `apps/cli/package.json@^5.6.2`). No `marked`, no `marked-terminal`, no new dependencies. Line-by-line transformation only — this is not a full Markdown parser.
- **Formatter rules** (exact, so QA can validate byte output):
  - `# text` → `chalk.bold.cyan.underline(text)` (level 1)
  - `## text` → `chalk.bold.cyan(text)` (level 2)
  - `### text` and lower (`####`, `#####`, `######`) → `chalk.bold(text)` (no color)
  - ` ``` ` fenced code block delimiters → `chalk.dim(line)`
  - Lines inside a fenced code block (between the two ` ``` ` markers) → `chalk.dim(line)`
  - `---` standalone horizontal rule → `chalk.dim('─'.repeat(60))`
  - `- item` or `* item` unordered list marker → replace marker with `chalk.cyan('•')`, preserve indentation and content
  - GFM table separator row (matches `/^\|[\s\-|]+\|$/`) → `chalk.dim(line)`
  - GFM table data row (starts and ends with `|` but isn't the separator) → pass through unchanged
  - Every other line → pass through unchanged
- **Pager integration** via `node:child_process.spawn`:
  - Use `$PAGER` if set (split on whitespace for arguments, e.g. `PAGER="less -R"` → `less` + `['-R']`)
  - Otherwise fall back to `less -R`
  - Write formatted text to the child's stdin, end stdin, await child exit
  - If pager spawn fails → fall back to direct stdout (not fatal, not an error)
- **Error handling** (all exit code 1):
  - Source not found in DB (no row matching the given UUID) → `printError('Source not found: <id>')`
  - Source exists but `layout.md` download fails (e.g. Extract never ran, or ran before spec 48 existed) → `printError('layout.md not found for source <id>. Run \`mulder extract <id>\` first.')`
  - GCP/cloud_sql config missing (and not in dev_mode) → `printError('GCP configuration with cloud_sql is required for show')` — matches the existing pattern from `extract` command

### Out of scope (explicit, to prevent creep)

- **No new dependencies** other than what's already in `apps/cli/package.json` (`chalk`, `commander`, `@mulder/core`, `@mulder/pipeline`, etc.)
- **No `--open` flag** that spawns `$EDITOR` — deferred (simple enough to add later if someone asks)
- **No `--json` output mode** — this is a display command, JSON makes no sense
- **No image/math/HTML rendering** in the Markdown
- **No full Markdown parser** — line-by-line regex transforms only. Complex edge cases (nested code blocks, inline HTML, reference links, etc.) pass through unchanged
- **No changes to spec 48's `layoutToMarkdown` or the Extract step**
- **No changes to CLAUDE.md, README, or other docs** outside the new spec file
- **No changes to `apps/api`** — the M7 API is a separate path, not this spec
- **No UI** — this is the stop-gap until M7-H11 delivers the real viewer
- **No removal** when M7-H11 lands — the CLI command can coexist with the web viewer

### Interfaces affected

- **New:** `apps/cli/src/commands/show.ts`
- **Modified:** `apps/cli/src/index.ts` (single import + single call)
- **Not modified:** `apps/cli/package.json` (no new deps), any other workspace package, database schema, config, prompts, fixtures

## 4. Blueprint

### 4.1 Files

1. **`apps/cli/src/commands/show.ts`** *(new, ~180-220 lines)*

   Exports `registerShowCommands(program: Command): void`. Internal helpers:
   - `formatMarkdown(text: string): string` — pure function, line-by-line ANSI transformation per the rules in §2 "Formatter rules". Only import from `chalk`. No I/O, no mutation.
   - `writeToPager(text: string): Promise<void>` — spawns `$PAGER` or `less -R`, streams the text, awaits exit.

   Imports:
   - `node:child_process` (spawn)
   - `@mulder/core`: `closeAllPools`, `createLogger`, `createServiceRegistry`, `findSourceById`, `getWorkerPool`, `loadConfig`
   - `chalk` (default import)
   - `commander` (`type Command`)
   - `../lib/errors.js` (`withErrorHandler`)
   - `../lib/output.js` (`printError`)

   Command definition (Commander):
   ```
   program
     .command('show')
     .description('Print the extracted layout.md for a source with terminal formatting')
     .argument('<source-id>', 'UUID of the source to show')
     .option('--raw', 'print plain Markdown (no ANSI colors)')
     .option('--pager', 'pipe formatted output through $PAGER (or `less -R`)')
     .action(withErrorHandler(async (sourceId, options) => { ... }));
   ```

2. **`apps/cli/src/index.ts`** *(modified, +2 lines)*

   Add alphabetically-ordered import:
   ```typescript
   import { registerShowCommands } from './commands/show.js';
   ```
   And the matching register call, alphabetically between `registerSegmentCommands` and `registerTaxonomyCommands`:
   ```typescript

   ```

### 4.2 Database Changes

None.

### 4.3 Config Changes

None. The command reads `config.gcp.cloud_sql` via `loadConfig()` exactly like every other data-accessing command.

### 4.4 Integration Points

- **CLI registration:** `apps/cli/src/index.ts` wires the new command into the Commander program alongside the existing 13 command groups. No order dependency.
- **Service abstraction:** `services.storage.download('extracted/{id}/layout.md')` returns a `Buffer` that `.toString('utf-8')` converts to the markdown text. No direct GCS SDK.
- **Source validation:** `findSourceById(pool, sourceId)` before attempting the download — catches bad UUIDs cleanly.
- **Error pattern:** `printError` + `process.exit(1)` for user-facing errors; `withErrorHandler` catches unexpected throws and prints stack traces.
- **Pool cleanup:** `closeAllPools()` in a `finally` block per the existing command pattern.

### 4.5 Implementation Phases

Single phase — this is one file plus a 2-line registration. Implement in one commit:

1. Create `apps/cli/src/commands/show.ts` with the full command definition, formatter, and pager helper.
2. Update `apps/cli/src/index.ts` with the import and register call.
3. Build and verify.

No need to split — the total code is small and cohesive.

## 5. QA Contract

The QA agent verifies this command without reading implementation code. Tests invoke the built CLI via `execFileSync` or `spawn`, capture stdout/stderr/exit code, and assert on observable bytes. The dev registry and an in-memory storage fixture (or a seeded `.local/storage/`) stand in for real GCP.

**Precondition for all QA-NN that need a real source:** a fixture test source is ingested and extracted before the test suite runs, producing a known UUID `SRC_ID` and a `.local/storage/extracted/$SRC_ID/layout.md` whose content is known. Use `fixtures/raw/native-text-sample.pdf` or equivalent.

### QA-01: Happy path with ANSI formatting
**Given** a source `SRC_ID` with a populated `layout.md` in storage,
**When** `mulder show $SRC_ID` runs,
**Then** the exit code is 0, stdout is non-empty, stdout contains the Markdown body text (e.g. `"Bundesamt für Flugsicherung"` for the native-text fixture), and stdout contains at least one ANSI escape sequence matching the chalk output for level-1 headings (the byte sequence `\x1b[` appears at least once).

### QA-02: `--raw` produces byte-identical Markdown with no ANSI
**Given** a source `SRC_ID` with `layout.md` content equal to the file at `.local/storage/extracted/$SRC_ID/layout.md`,
**When** `mulder show $SRC_ID --raw` runs,
**Then** the exit code is 0 and stdout (after trimming a trailing newline, if the command adds one) is byte-identical to the raw file content. Stdout contains zero `\x1b[` bytes (no ANSI escape sequences).

### QA-03: `--help` prints usage information
**Given** the CLI is built,
**When** `mulder show --help` runs,
**Then** the exit code is 0, stdout contains the word `Usage:` (Commander's standard help header), the substring `show <source-id>` (the command signature), the substring `--raw`, the substring `--pager`, and a human-readable description of each flag.

### QA-04: Nonexistent source ID → clear error, exit 1
**Given** a well-formed UUID that does not match any row in the `sources` table (e.g. `00000000-0000-0000-0000-000000000000`),
**When** `mulder show 00000000-0000-0000-0000-000000000000` runs,
**Then** the exit code is 1 and stderr contains the substring `Source not found` and the substring `00000000-0000-0000-0000-000000000000`.

### QA-05: Source exists but `layout.md` missing → clear error, exit 1
**Given** a source that has been ingested but not extracted (so `sources.status = 'ingested'`, no `extracted/{id}/layout.md` file),
**When** `mulder show $SRC_ID` runs,
**Then** the exit code is 1 and stderr contains the substring `layout.md not found` and the substring `mulder extract` (the hint to run extract first).

### QA-06: Formatter is deterministic (idempotency)
**Given** the same source `SRC_ID` and identical stored `layout.md`,
**When** `mulder show $SRC_ID` runs twice,
**Then** both runs produce byte-identical stdout. No timestamps, no run-specific IDs, no system-dependent state leak into the output.

### QA-07: `--pager` does not crash and produces output
**Given** a source `SRC_ID` and `PAGER=cat` in the environment (so the pager is a no-op passthrough),
**When** `mulder show $SRC_ID --pager` runs,
**Then** the exit code is 0 and stdout contains the formatted Markdown body (the pager passed it through). The test is compatible with environments where `less` may not be installed — `PAGER=cat` forces the fallback path.

### QA-08: GFM table separator row is dimmed
**Given** a source whose `layout.md` contains a GFM table (e.g. the `table-layout-sample` fixture run through Extract),
**When** `mulder show $SRC_ID` runs (without `--raw`),
**Then** stdout contains the ANSI dim sequence (`\x1b[2m` or `\x1b[22m`) on the table separator row line (`| --- | --- | --- | --- |` becomes dimmed) while the data rows remain un-dimmed (no `\x1b[2m` immediately preceding `| Bayern |`).

## 6. Cost Considerations

**None — no paid API calls.** The command reads a local file via `services.storage.download()` (which in dev mode reads from `.local/storage/`, and in production would read from GCS — both are cheap reads, no LLM or OCR involved). The formatter is pure TypeScript string manipulation. No Gemini, no Document AI, no embeddings, no external HTTP.

## 7. Known limitations (shipped intentionally)

1. **Not a full Markdown parser.** Line-by-line regex transforms only. Edge cases like nested code blocks, inline HTML, reference-style links, or complex tables with inline formatting will pass through plain. This is intentional — a full parser is what `marked-terminal` would provide, and we're deliberately not taking that dependency.
2. **All `###` and deeper headings look the same.** Levels 3-6 all render as `chalk.bold()`. Adequate for spec 48's output, which almost never goes deeper than level 2. Hierarchical differentiation is a V2 concern.
3. **No syntax highlighting inside code blocks.** Code blocks are dimmed as a whole; individual tokens are not colored. Out of scope.
4. **No image placeholders.** `![alt](url)` passes through unchanged. Terminals can't render images anyway; a placeholder would be noise.
5. **`--pager` crashes recover silently.** If `spawn` fails (no pager binary, permission denied, etc.), the command falls back to direct stdout without emitting a warning. The user gets the output either way; a warning would be noise.
6. **The command will coexist with the M7-H11 web viewer indefinitely.** This is not a throwaway — CLI preview is useful even when a web UI exists (e.g. for remote SSH sessions, CI logs, quick checks). Spec 49 does not need to be deleted or deprecated when H11 lands.

## 8. Architecture alignment

- **Service abstraction:** `services.storage.download()` goes through the dev/GCP service registry, never the GCS SDK directly. Matches CLAUDE.md "Service Abstraction" rules.
- **Config via loader:** `loadConfig()` — no direct YAML parsing.
- **Custom error messages via `printError`:** no generic `throw new Error()`. The command exits cleanly with `process.exit(1)` after printing.
- **ESM, strict mode, no `any`/`as`:** the command file has fully-typed arguments and return types. Only `unknown` catches, per convention.
- **Logging:** via `createLogger()` + service registry. No `console.log`.
- **Pattern match:** the file structure mirrors `apps/cli/src/commands/extract.ts` so closely that a reader familiar with one will immediately understand the other.
